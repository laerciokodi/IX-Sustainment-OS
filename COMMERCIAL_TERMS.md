## Purpose

This document explains the intended **commercial posture** for IX Sustainment OS.

It exists to make the monetization and deployment boundary explicit for:
- evaluators
- pilot customers
- enterprise buyers
- government-adjacent users
- integrators
- partners
- legal and procurement reviewers

This file is an **informational commercial-terms overview** only.

It is **not** the final negotiated order form, master services agreement, government contract vehicle, or legal opinion. Any binding commercial relationship must be established through a separately executed written agreement.

---

## Licensing posture

IX Sustainment OS is published under the **Business Source License 1.1 (BSL 1.1)** in this repository.

That means public source access is available, but **production or commercial use is not automatically granted** by default.

The public repository is intended to support:
- review
- technical evaluation
- non-production experimentation
- architecture assessment
- limited internal feasibility work consistent with the repository license

The public repository is **not** intended to grant unrestricted production deployment rights.

---

## When a commercial license is required

A separate commercial license is expected for any use that includes one or more of the following:

- production deployment
- operational deployment in a government or government-adjacent environment
- paid pilot use
- live operational data use
- internal enterprise use beyond evaluation
- managed-service or hosted-service delivery
- deployment for a customer, client, or third party
- reseller or OEM packaging
- integration into a broader commercial platform
- custom support, feature development, or deployment assistance

If an organization wants to do more than evaluate the code in a non-production manner, it should expect to enter a commercial agreement.

## Commercial offer structure

The intended commercial structure has four lanes.

### 1. Evaluation access
For technical review and non-production validation.

Typical scope:
- internal review
- architecture validation
- proof-of-concept examination
- lab-only exploration
- security review of public code

Typical characteristics:
- no production rights
- no SLA
- no deployment support obligation
- no warranty
- no indemnity
- no guaranteed roadmap commitments

### 2. Paid pilot
For limited operational trial use in a bounded environment.

Typical scope:
- specific site, team, or use case
- fixed duration
- bounded user population
- bounded data sources
- bounded support window
- success criteria defined up front

Typical characteristics:
- paid limited-use rights
- pilot support terms
- defined implementation boundary
- agreed reporting cadence
- written success/failure criteria
- no implied perpetual rights after pilot expiry

### 3. Production license
For real operational deployment.

Typical scope:
- internal organizational use
- program-level or enterprise-level deployment
- live workflow integration
- controlled environment configuration
- supported releases
- contracted support or upgrade rights

Typical characteristics:
- production-use rights
- defined deployment model
- defined scope of authorized users or instances
- optional maintenance and support
- optional training and onboarding
- optional security review support
- optional deployment engineering support

### 4. Strategic / government / OEM agreement
For larger-scale or more sensitive use.

Typical scope:
- major deployment programs
- integrator partnerships
- OEM redistribution
- government-adjacent operational programs
- private-hosted or enclave-hosted deployments
- custom roadmap or integration commitments

Typical characteristics:
- negotiated rights
- negotiated support levels
- deployment-specific boundaries
- tailored data-handling and security terms
- custom acceptance and delivery terms
- optional source-escrow or continuity terms if negotiated

## Intended pricing logic

This repository does **not** publish fixed public pricing.

Commercial pricing is intended to reflect:
- deployment scope
- number of users or teams
- number of operational sites
- integration complexity
- hosting model
- support requirements
- required hardening
- compliance or documentation burden
- training needs

## Government and government-adjacent use

IX Sustainment OS is being framed as a **non-weapon, defense-adjacent sustainment platform**.

It is intended for categories such as:
- maintenance operations
- readiness management
- sustainment coordination
- technical-data entitlement workflows
- parts bottleneck visibility
- auditable human-governed recommendation support

It is **not** offered as:
- a weapon system
- a targeting system
- a strike-planning platform
- an autonomous engagement platform
- a kill-chain optimization product
- a battle-damage exploitation or fires-control system

Any government or government-adjacent use is expected to be governed by a separate written commercial agreement with clearly defined deployment and rights boundaries.

- auditability
